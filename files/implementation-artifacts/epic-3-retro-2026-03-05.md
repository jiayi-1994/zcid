## 挑战与改进

### 1. Casbin g2 策略未集成
**现状:** 架构文档规划了 Casbin g2 三元组用于项目级角色管理，但当前 RBAC enforcer 不包含 g2 定义。
**应对:** 使用 project_members 表替代，将 Casbin g2 集成推迟。
**影响:** 低。project_members 表方案功能完整，g2 集成可作为后续优化。
**建议:** 在 Epic 后续阶段或专门的技术债还清 Sprint 中补充 Casbin g2 集成。

### 2. Handler 层权限检查粒度
**现状:** 环境和服务的 handler 初始实现仅检查 admin 角色，不符合 AC 要求的 "管理员或项目管理员"。
**根因:** 复制 project handler 的权限逻辑时未根据 story AC 调整。
**修复:** Code review 阶段修正，添加 `isAdminOrProjectAdmin` 辅助函数。
**建议:** 后续 story 实现时，先审查 AC 中的权限要求再编码。

### 3. 项目级资源访问无中间件校验
**现状:** 环境和服务的 API 端点只通过 JWT 认证，未验证用户是否有权访问特定项目。
**风险:** 任何已认证用户可访问任何项目的子资源。
**建议:** Epic 4 开始前，添加项目级访问中间件或在 service 层增加项目权限校验。

### 4. Go 包命名冲突
**现状:** 服务管理模块不能使用 `internal/service/`（与 Go 的 service 层概念冲突），改用 `internal/svcdef/`。
**影响:** 低。命名清晰且不影响功能。
**建议:** 在 project-context 中记录此命名约定。

## 关键洞察

1. **模式复用加速开发:** 建立良好的模块模板（handler→service→repo + 统一错误处理）后，后续同类模块可快速复制并适配。
2. **Code Review 不可省略:** 对抗性审查发现了类型安全和权限模型两个高严重度问题，均在合并前修复。
3. **前瞻性设计有回报:** project_members 表提前创建的决策在 3.4 中直接受益。
4. **AC 驱动开发:** 权限检查问题暴露了"复制粘贴时忽略 AC 差异"的风险。

- Epic 3 的 projects 表和 project_members 表是 Epic 4 的基础

- 密钥加密需要新的 AES 加密工具包
- K8s Secret 注入需要 K8s client 集成（新技术栈）
