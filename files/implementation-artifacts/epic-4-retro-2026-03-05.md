## 成功与亮点

### 1. 模式复用再次加速开发
handler→service→repo 三层模式继续被高效复用。variable 模块从零到完整只用了一轮迭代，9 个 API 端点一次性实现正确。证明 Epic 3 建立的模块模板已完全成熟。

### 2. 安全设计决策合理
AES-256-GCM 加密方案经过充分考量:
- 密钥通过环境变量注入 (ZCID_ENCRYPTION_KEY)，不进入代码仓库
- nonce + ciphertext + tag 组合存储，每次加密生成不同密文
- 未设置密钥时服务正常启动但密钥功能不可用，不影响非密钥场景

### 3. 数据库设计精巧
partial unique index (`WHERE status != 'deleted'`) 优雅解决了软删除与唯一约束的冲突。全局变量和项目变量分别用独立的 partial unique index，避免跨作用域冲突。

### 4. 跨 Epic 技术债闭环
本 Epic 完成了 Epic 2 遗留的 FR5 (密钥变量对普通成员不可见)，通过 `FilterForRole` 在 service 层过滤后端数据。这证明了合理的技术债管理策略: 在正确的时机（变量模块实现时）闭环跨 Epic 依赖。

## 挑战与改进

### 1. Code Review 发现 total 计数不一致 (HIGH)
**现状:** `ListProjectVariables` 在 FR5 过滤后返回的 `total` 仍是过滤前的值，导致客户端分页显示与实际条目数不一致。
**根因:** 过滤逻辑在 handler 层执行，但 total 在过滤前已确定。
**修复:** Code review 阶段修正，total 改为 `len(items)` 过滤后值。
**教训:** 分页 + 过滤场景下，total 必须在过滤后重新计算。

### 2. 错误码重复定义 (MEDIUM)
**现状:** `CodeVarDuplicate` 和 `CodeDecryptFailed` 同时在 `internal/variable/service.go` 和 `pkg/response/codes.go` 中定义。
**根因:** 开发时先在 service 中定义常量，后续统一迁移到 codes.go 时遗忘清理。
**修复:** 移除 service.go 中的重复定义，统一引用 `response` 包。
**教训:** 错误码必须有唯一权威来源 (pkg/response/codes.go)。

### 3. 项目级访问控制缺失延续 (P0 - 本次修复)
**现状:** Epic 3 回顾已标记为高优技术债，但 Epic 4 开发期间未处理，导致变量 API 也继承了此漏洞。
**根因:** Epic 之间的行动项缺乏强制执行机制。
**修复:** 本次回顾时实施 `RequireProjectScope` 中间件，覆盖所有 `/:id/*` 子资源路由。
**教训:** P0 安全问题必须在下一个 Epic 开始前强制修复，不可推迟。

### 4. 流水线级变量未实现
**现状:** Epic 定义了全局/项目/流水线三级变量，但当前仅实现全局和项目两级。
**应对:** 流水线级变量依赖流水线模块 (Epic 6)，故推迟是合理的。
**影响:** 低。合并查询逻辑已预留扩展位。

## 关键洞察

1. **P0 问题必须阻断式修复:** 项目级访问控制漏洞从 Epic 3 延续到 Epic 4，影响面持续扩大。回顾驱动的 P0 修复比计划修复更紧急但更被动。建议: Epic 完成时立即修复 P0 后才能标记 epic-done。

2. **partial unique index 是软删除的最佳搭档:** 在 PostgreSQL 中，`WHERE status != 'deleted'` 的条件唯一索引完美解决了"软删除后允许重建同名记录"的需求，无需业务层额外校验。

3. **环境变量密钥管理是安全的起点:** AES-256-GCM + 环境变量方案简单有效，但生产环境应考虑 Vault/KMS 集成。当前方案作为 MVP 足够。

4. **Stub 接口设计加速交付:** SecretInjector 接口定义让 Story 4.3 在无 K8s 环境下完成，同时为 Epic 7 提供了清晰的集成点。

5. **错误码集中管理:** 错误码重复定义的教训表明，所有业务错误码必须在 `pkg/response/codes.go` 统一定义，业务模块只引用不定义。

## P0 修复记录

### RequireProjectScope 中间件 (本次回顾实施)

**问题:** 任何已认证用户可通过 JWT 访问任何项目的子资源 (环境、服务、成员、变量)。

**修复方案:**
1. 新增 `pkg/middleware/project_scope.go`: 检查 JWT 用户是否为项目成员或系统 admin
2. 拆分项目 handler 路由注册: `RegisterCollectionRoutes` (无需 scope) + `RegisterResourceRoutes` (需要 scope)
3. 在 `main.go` 中创建 `projectScope` 路由组 (`/:id`)，应用 `RequireProjectScope` 中间件
4. 所有子资源路由 (environments, services, members, variables) 挂载在 `projectScope` 下

**影响范围:**
- `pkg/middleware/project_scope.go` (新建)
- `internal/project/handler.go` (路由注册拆分)
- `cmd/server/main.go` (路由结构调整)

**验证:** 全部后端测试通过 (auth, environment, project, svcdef, variable)，前端 16 测试通过。
