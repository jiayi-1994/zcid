# Product Brief: zcid

## Executive Summary

zcid 是一个基于云原生技术栈（Tekton + ArgoCD）构建的一站式 CI/CD 平台，目标是为开发者和 DevOps 工程师提供「小白都好用」的持续集成与持续部署体验。平台采用 Apple 风格蓝白色调的现代化 UI，将 Tekton 和 ArgoCD 的强大能力封装在直观的可视化操作界面之后，让用户无需编写 YAML 即可完成从代码提交到生产部署的全流程。

zcid 定位为可商业化产品，支持私有云独立部署和 PaaS 平台模块化嵌入两种交付模式，填补市场上「云原生 CI/CD 平台体验好且可私有化部署」的空白。

---

## Core Vision

### Problem Statement

Kubernetes 已成为容器编排的事实标准，Tekton 和 ArgoCD 分别是 CI 和 CD 领域的云原生标杆工具。然而，它们的原生界面完全面向 YAML 编辑和命令行操作，对于绝大多数开发者和 DevOps 初学者来说，使用门槛极高，操作体验极差，尤其不符合国内用户的操作习惯。

### Problem Impact

- **开发效率低下：** 开发者花大量时间在编写和调试 CI/CD YAML 配置上，而非核心业务代码
- **学习曲线陡峭：** Tekton Pipeline/Task/Step 三层嵌套 + ArgoCD Application 模型，初学者需要数周才能上手
- **运维负担重：** 缺少统一管理面，流水线配置、环境管理、镜像仓库对接、日志查看分散在多个系统中
- **企业交付困难：** 企业需要可商业化、可私有部署的 CI/CD 方案，但现有开源产品体验不够好，云厂商方案又锁定厂商

### Why Existing Solutions Fall Short

- **Tekton/ArgoCD 原生 UI：** 功能强大但交互原始，纯 YAML 操作对非专家用户不友好
- **Zadig：** 功能全面但社区活跃度下降，UI 体验一般，定制灵活度不足
- **Jenkins：** 用户基数大但架构老旧，非云原生设计，K8s 集成笨重，无法跟上云原生趋势
- **阿里云效/云原生应用平台：** 体验好但深度锁定云厂商，无法私有化部署，不适合作为 PaaS 模块集成出售
- **GitLab CI/CD：** 代码仓库一体化但 CD 能力弱，不是真正的云原生编排

### Proposed Solution

zcid 作为 Tekton + ArgoCD 之上的智能翻译层和状态看板，将用户意图翻译成 Kubernetes 资源，同时提供实时状态监控和全流程可视化：

- **可视化流水线编排：** 拖拽式 Stage→Step 编排，模板一键创建常见 CI/CD 场景，高级用户可切换 YAML 模式
- **统一管理面：** 项目 → 环境 → 服务三层资源模型，一个界面管理代码构建、镜像推送、环境部署全流程
- **实时反馈：** WebSocket 实时构建日志推送，流水线状态即时更新
- **极简运维：** PostgreSQL + Redis + MinIO 三组件覆盖全部基础能力，K8s 中部署仅需 3 个 StatefulSet
- **安全合规：** 四级变量体系 + AES-256 凭证加密 + RBAC 权限控制 + 操作审计

### Key Differentiators

1. **「小白都好用」的产品理念：** 不暴露 Tekton/ArgoCD 复杂性，三层用户（新手模板/中级可视化/高级 YAML）一个入口三种体验
2. **真正的云原生架构：** 不是套壳 Jenkins，底层 Tekton+ArgoCD 确保与 K8s 生态完全兼容
3. **极简中间件栈：** 比 Zadig 少两个中间件组件，运维复杂度砍半
4. **双模交付：** 私有云独立部署 + PaaS 模块化嵌入，覆盖自用和商业化两个场景
5. **市场空白填补：** 目前市场上没有「云原生 + 体验好 + 可私有部署 + 可商业化」四合一的 CI/CD 方案
6. **现代化设计语言：** Apple 风格蓝白色调，符合国人操作习惯，与同类产品形成明显的视觉差异化

## Target Users

### Primary Users

**1. 后端开发者（代表：小李）**

- **画像：** 25 岁，3 年经验，Go/Java 后端，对 K8s 有基础了解但不深入
- **核心需求：** 提交代码后直观看到构建进度和日志，不需要理解 Tekton/ArgoCD 底层
- **使用频率：** 每日多次（提交代码、查看构建、确认部署）
- **痛点：** 被要求写 Tekton YAML 完全不会，每次改流水线都要找 DevOps
- **成功标志：** 不用找人帮忙就能自己跑通构建和部署

**2. DevOps 工程师（代表：老张）**

- **画像：** 30 岁，5 年经验，熟悉 K8s/Helm/CI/CD 全流程，团队基础设施负责人
- **核心需求：** 高效配置和管理 CI/CD 全流程，同时赋能开发团队自助操作
- **使用频率：** 每日（配置管理、异常处理、环境维护）
- **痛点：** 直接操作 CRD+CLI 效率尚可，但无法赋能开发者，自己成了团队瓶颈
- **成功标志：** 开发者不用再找自己配流水线，只处理真正需要运维介入的事

**3. 技术团队 Leader / CTO（代表：王总）**

- **画像：** 35 岁，技术管理者，关注团队效率和交付质量
- **核心需求：** 全局 Dashboard 一眼看到构建趋势、部署频率、环境健康状况
- **使用频率：** 每日/每周（查看全局状态、支撑技术决策）
- **痛点：** 没有统一视图，要问 DevOps 才知道各环境状态
- **成功标志：** 不用问人就知道团队的 CI/CD 健康状况

### Secondary Users

**4. 系统管理员**

- **画像：** 通常由 DevOps 工程师兼任或独立平台运维角色
- **核心需求：** 简洁的系统设置（集群连接、镜像仓库、用户权限、全局变量、通知渠道）
- **权限模型：** 管理员 / 项目管理员 / 普通成员 三级 RBAC（保持简单）

### User Journey

**开发者旅程（小李）：**

1. **发现：** DevOps 在团队内推广 zcid，邀请加入项目
2. **上手：** 登录后看到 Dashboard，项目和环境一目了然
3. **核心使用：** 提交代码 → Webhook 自动触发 → 实时日志 → 自动部署 → 通知到达
4. **Aha 时刻：** 第一次全程不写 YAML 完成从提交到部署的闭环
5. **进阶：** 学会从模板创建流水线，偶尔自定义步骤

**DevOps 旅程（老张）：**

1. **发现：** 评估替代 Jenkins/原生 Tekton 的方案，试用 zcid
2. **上手：** 系统设置 → 对接 Git/Harbor/集群 → 创建第一个项目和流水线
3. **核心使用：** 模板+可视化编排配置流水线，管理多环境，赋能开发者
4. **Aha 时刻：** 开发者第一次自己完成了流水线配置和部署
5. **长期：** 专注高级运维（YAML 模式微调、集群管理、性能优化），日常 CI/CD 操作下放给开发者

## Success Metrics

### 用户成功指标

**核心链路验证：**

- **容器化微服务全链路：** 用户能完成 代码拉取 → 构建 → 镜像推送到 Harbor → 部署到 K8s 环境 的完整闭环
- **传统应用构建链路：** 用户能完成 代码拉取 → 编译打包（如 JAR/WAR）→ 上传到 MinIO 对象存储 的完整闭环
- **多服务并行构建：** 一个项目下多个微服务能同时触发构建，互不干扰
- **流水线自助率：** 开发者能自己从模板创建流水线并成功运行，不需要 DevOps 介入

**体验指标：**

- **上手时间：** 新用户从登录到第一次成功跑通流水线 < 30 分钟
- **日常操作效率：** 查看构建日志、触发构建、确认部署状态均可在 3 次点击内完成
- **构建可视性：** 用户能实时看到每个 Step 的执行状态和日志输出

### Business Objectives

**当前阶段目标（产品优先，商业后续）：**

- **3 个月目标：** P0 核心链路 MVP 完成，内部团队可用，容器化 + 传统应用两条链路跑通
- **6 个月目标：** P1 功能完善（ACR、多架构构建、Cron 触发），具备对外展示和试用的产品完整度
- **12 个月目标：** PaaS 嵌入能力就绪（OAuth 集成），可以作为模块进入商业化流程

### Key Performance Indicators

| KPI | 衡量方式 | 目标 |
|-----|---------|------|
| 核心链路成功率 | 流水线执行成功率 | > 95% |
| 多服务支撑能力 | 单项目支持的微服务数量 | 无上限（受集群资源限制） |
| 构建产物覆盖 | 支持的产物类型（镜像/JAR/前端包等） | MVP 至少覆盖镜像 + JAR 两种 |
| 用户自助率 | 开发者自己创建并成功运行流水线的比例 | > 70% |
| 平台稳定性 | 月可用率 | > 99.5% |
| 部署响应时间 | 从触发到 ArgoCD 开始同步 | < 30 秒 |

## MVP Scope

### P0 核心功能（MVP）

MVP 聚焦打通两条核心构建链路的完整闭环，按交付波次组织：

**Wave 1 — 基础设施层：**

1. **用户认证与权限管理：** JWT 无状态认证 + Casbin 三级 RBAC（管理员/项目管理员/普通成员）
2. **项目/环境/服务 CRUD：** Project → Environment（K8s Namespace）→ Service 三层资源模型基础管理
3. **变量与密钥管理：** 四级变量体系（全局→项目→流水线→运行时），密钥变量 AES-256-GCM 加密存储，运行时临时 K8s Secret 注入
4. **Git 仓库集成：** GitProvider 接口，GitLab + GitHub OAuth 授权关联，代码仓库浏览与分支选择

**Wave 2 — 核心链路（CI）：**

5. **流水线可视化编排 + 模板系统：** 简化 Stage→Step 可视化编排；提供至少 4 个高质量模板（Go 微服务构建、Java Maven 构建、前端 Node 构建、通用 Docker 构建）；高级用户可切换 YAML 模式
6. **Tekton PipelineRun 翻译与执行：** 将用户可视化编排翻译为 Tekton CRD 并提交执行；MVP 先支持有限 Step 类型（git-clone、build、push-image、upload-artifact、deploy）
7. **实时构建日志：** WebSocket + client-go Pod 日志直读（Follow 模式）；MVP 简化为单连接 Watch 单 Pod，并发 Watch 留后续优化
8. **Kaniko 镜像构建 + Harbor 推送：** 容器化链路（代码拉取→构建→Kaniko 打镜像→推送 Harbor）
9. **传统应用构建：** 传统链路（代码拉取→编译打包 JAR/WAR→上传 MinIO 对象存储）

**Wave 3 — 闭环体验（CD + 运营）：**

10. **ArgoCD 部署 + 状态展示 + 回滚：** 创建 ArgoCD Application + 手动同步 + 部署状态实时展示 + 部署历史回滚；自动同步策略放 P1
11. **Webhook 触发：** 复用 Tekton Triggers EventListener，支持 GitLab/GitHub Push/MR 事件自动触发流水线
12. **通知：** Notifier 接口，邮件 + Webhook 通知（构建成功/失败、部署完成）
13. **审计日志（简化版）：** Gin 中间件拦截写操作，记录「谁+时间+操作+目标+结果」到 PostgreSQL 审计表，不含复杂查询界面
14. **简化版 Dashboard：** 项目状态一览（最近构建列表、部署状态、流水线运行情况），不含趋势图和热力图

### MVP 关键设计约束

- **Tekton 翻译层**是核心技术难点，需在实现前进行专项设计评审
- **错误状态体验**必须在 MVP 中做好——构建失败、部署失败的日志定位和展示是高频场景
- **模板质量**决定第一印象——模板必须「选一个→改几个参数→跑起来」，降低用户上手门槛

### Out of Scope（明确不做）

**P1 — 完善阶段：**

- 阿里云 ACR 镜像仓库支持（RegistryProvider 接口已预留）
- BuildKit amd64+arm64 多架构构建
- Cron 定时触发
- 飞书/钉钉/企业微信通知（Notifier 接口已预留）
- 多集群管理（ClusterManager 接口已预留）
- OAuth 集成（PaaS 平台嵌入场景）
- ArgoCD 自动同步策略
- WebSocket 并发多 Pod 日志 Watch
- 多语言支持（中英文）

**P2 — 增强阶段：**

- 镜像变更触发 CD
- Dashboard 高级数据可视化（趋势图、热力图）
- 流水线模板市场
- 全局搜索

**明确不做：**

- 不自建任务调度引擎（Tekton 负责）
- 不自建部署编排（ArgoCD 负责）
- 不引入 ELK/Loki（Pod 日志 + MinIO 够用）
- 不引入独立消息队列（Redis Streams 够用）
- 不拆微服务（管理平台单体够用）

### MVP 成功标准

- 容器化微服务全链路跑通：代码拉取 → 构建 → Kaniko 打镜像 → 推送 Harbor → ArgoCD 部署到 K8s
- 传统应用构建链路跑通：代码拉取 → Maven/Gradle 编译 → JAR/WAR 上传 MinIO
- 开发者能从模板自助创建流水线并成功运行
- 新用户上手时间 < 30 分钟
- 构建日志实时可见，失败原因可快速定位

## Future Vision

### P1 路线图（6 个月目标）

完善平台功能，达到对外展示和试用的产品完整度：

- **镜像仓库扩展：** 阿里云 ACR 支持，覆盖公有云场景
- **多架构构建：** BuildKit 支持 amd64+arm64 双架构镜像，跟上行业主流
- **自动化增强：** Cron 定时触发、ArgoCD 自动同步策略
- **协作增强：** 飞书/钉钉/企业微信通知，覆盖国内主流 IM
- **多集群管理：** 支持管理多个 K8s 集群，满足多环境隔离需求

### P2 路线图（12 个月目标）

- **PaaS 嵌入就绪：** OAuth 集成，可作为模块嵌入公司 PaaS 平台进行商业化
- **数据驱动运营：** Dashboard 高级可视化（构建趋势图、部署频率热力图）
- **生态扩展：** 流水线模板市场、全局搜索、镜像变更触发 CD

### 长期愿景

zcid 成为国内云原生 CI/CD 领域的标杆产品，以「小白都好用」的产品体验 + 真正云原生的技术架构 + 灵活的私有化部署能力，服务从初创团队到大型企业的全谱系用户。
